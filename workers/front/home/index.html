One store.

Project Matrix is one vinyl store — %%SNAP_COUNT%% real Discogs releases, frozen %%SNAP_DATE%% — being built five ways: a different rendering architecture under each build, identical data and pixels by construction, one pinned measurement ruler over all of it.

The premise under test: no architecture wins everywhere. The numbers exist to show where each one earns its bytes, where the same one is overkill, and what both cost — in performance, UX, and infra spend.

The instrument shipped first — no verdict can be retrofitted. Live today: the frozen snapshot, the composed origin, the drift gate, the bench harness. The store surfaces are in build; each catalogue entry below links its decision record, and every number published here will be a URL you can re-run.

Why “Matrix”

The matrix number is the code etched in a record’s runout groove, naming the master and cut it was pressed from. Same album, different cut — which sounds better depends on the system you play it on. Same store, different architecture: which fits is what the numbers are for.

The method

How the numbers stay honest

  • Frozen snapshot

    Real data captured once from %%SNAP_SOURCE%%, dated, pinned to commit %%SNAP_SHA7%% — served the way production serves cacheable data. The manifest is live.

  • One ruler

    The same pinned web-vitals build measures every page, lab and field, and bytes are counted off the wire. The ruler itself.

  • The URL is the measurement condition

    Path carries the architecture; query carries the environment. A URL is a complete, reproducible receipt. ADR‑0004.

  • Held constant by construction

    Same host (one plane), same data (frozen snapshot), same pixels (a drift gate in CI) — so a gap can only be the paradigm. ADR‑0003.
